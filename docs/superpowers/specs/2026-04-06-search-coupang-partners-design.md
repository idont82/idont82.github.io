# /search/ 쿠팡 파트너스 콘텐츠 사이트 설계

- **날짜:** 2026-04-06
- **위치:** `idont82.github.io/search/`
- **상태:** 초안 (사용자 리뷰 대기)

## 1. 목적

`idont82.github.io/search/` 하위에 **SEO로 유입을 받아 쿠팡 파트너스 링크로 전환**시키는 독립 콘텐츠 사이트를 운영한다. 기존 인형뽑기 가이드(`index.html`)와는 도메인만 공유할 뿐 주제·디자인·탐색 동선이 완전히 분리된다.

성공 지표는 구글/네이버 검색 유입 → 쿠팡 파트너스 링크 클릭이다.

## 2. 핵심 의사결정

| 항목 | 결정 | 근거 |
|---|---|---|
| 콘텐츠 포맷 | 카테고리별 **TOP 3 비교** 페이지 | TOP 10보다 제작 부담 1/3, 단일 리뷰보다 키워드 확장성·전환율 높음 |
| 주제 범위 | 인형뽑기와 무관한 독립 니치 | 사용자 결정. 구체 카테고리는 첫 글 작성 시 정함 |
| 데이터 저장 | **HTML inline** (JSON/빌드 없음) | SEO 최강. 구글이 JS 렌더링 콘텐츠를 처리할 때 지연·누락 리스크 회피. 기존 프로젝트 "no build step" 원칙 유지 |
| URL 슬러그 | **영문** (예: `diapers-top3.html`) | Git/Windows/에디터에서 파일 관리 용이. 2026년 구글은 URL 언어에 거의 영향 없음. 키워드 매칭은 title·h1·본문이 담당 |
| 시각적 분리 | 기존 사이트와 **완전 분리** | 쿠팡 파트너스는 주제 혼합에 민감. 타겟 방문자가 전혀 다름. 다크 아케이드 테마는 파트너스 콘텐츠의 신뢰감에 부정적 |
| 기술 스택 | 바닐라 HTML/CSS, 빌드 없음 | 기존 프로젝트(`CLAUDE.md`) 원칙 준수 |

## 3. 디렉토리 구조

```
search/
├── index.html                  # 허브 페이지: 발행된 글 목록
├── assets/
│   ├── style.css               # 공통 스타일 (모든 글에서 <link>로 참조)
│   └── partners.js             # (선택) 링크 클릭 트래킹용, 없어도 동작
├── template.html               # 새 글 작성용 복사 템플릿 (참고용)
└── <slug>-top3.html            # 글 하나 = 파일 하나 (예: diapers-top3.html)
```

### 공통 리소스 공유 방식

- **CSS**: `assets/style.css` 한 곳에 두고 모든 페이지가 `<link>`로 참조. 한 번 수정하면 전체 반영.
- **헤더/푸터 마크업**: 작고 거의 안 바뀌므로 각 HTML에 복붙. JS include는 SEO 본문에 쓰지 않는다 (크롤러의 JS 렌더링 지연 리스크).
- **새 글 작성 워크플로**: `template.html`을 복사 → 파일명을 `<slug>-top3.html`로 변경 → 내용 교체 → `index.html` 목록에 한 줄 추가 → `sitemap.xml`에 URL 추가.

## 4. 페이지 템플릿 (TOP 3 비교)

모든 콘텐츠는 HTML에 직접 작성된다. 페이지 구조는 아래 순서를 따른다.

### 4.1 `<head>` 필수 요소

- `<title>`: 핵심 키워드 + "비교 TOP 3 추천" + 연도 포함
- `<meta name="description">`: 150자 이내, 핵심 키워드 포함
- `<link rel="canonical">`: 절대 URL
- Open Graph 태그 (`og:title`, `og:description`, `og:image`, `og:url`)
- `<script type="application/ld+json">`: `ItemList` 스키마 (3개 제품)
- `<script type="application/ld+json">`: `FAQPage` 스키마 (FAQ 섹션 대응)

### 4.2 `<body>` 섹션 순서

1. **쿠팡 파트너스 고지문구** (상단 띠, 의무)
2. **H1 + 공개일/갱신일**
3. **인트로** (3~5줄): 이 카테고리를 고를 때 봐야 할 기준
4. **요약 비교표**: 3열 × 핵심 스펙 행
5. **🥇 1위 섹션**
   - 제품 이미지 (alt에 제품명 포함)
   - 한줄평
   - 장점 / 단점
   - 주요 스펙
   - "쿠팡에서 보기" CTA 버튼 → 파트너스 링크
6. **🥈 2위 섹션** (동일 구조)
7. **🥉 3위 섹션** (동일 구조)
8. **선택 가이드** (2~3문단): 어떤 사람에게 어떤 제품을 추천하는지
9. **FAQ 2~3개**
10. **쿠팡 파트너스 고지문구** (하단 재노출)

### 4.3 SEO 요구사항

- `<title>`, `<h1>`에 핵심 키워드 ("○○ 추천", "○○ 비교", "TOP 3", 연도) 포함
- `ItemList` + `FAQPage` JSON-LD 구조화 데이터 (구글 rich result 대상)
- 모든 이미지에 `alt` 속성, 제품명 포함
- 쿠팡 파트너스 링크는 `rel="sponsored nofollow"` + `target="_blank"` (구글 정책 + 사용자 이탈 방지)

## 5. 쿠팡 파트너스 컴플라이언스

모든 페이지에 아래 항목을 강제한다.

1. **고지문구 상·하단 이중 노출**
   - 정확한 문구: `이 포스팅은 쿠팡 파트너스 활동의 일환으로, 이에 따른 일정액의 수수료를 제공받습니다.`
2. **모든 쿠팡 링크 속성**: `rel="sponsored nofollow"` + `target="_blank"`
3. **금지 표현**: "최저가 보장", "100% 환불", 허위·과장 표현 금지
4. **상표 사용**: 쿠팡 로고·상표 무단 사용 금지. CTA는 "쿠팡에서 보기" 같은 텍스트로만 구성
5. **색인 허용**: `robots.txt`는 `/search/` 크롤링 허용 상태 유지
6. **사이트맵 등록**: `sitemap.xml`에 각 페이지 URL 등록

## 6. 기존 사이트와의 간섭 방지

- 기존 `index.html`(인형뽑기 가이드)의 헤더/푸터/네비/해시 라우터에는 **어떤 변경도 없다**
- `/search/` 페이지에서 기존 사이트로 돌아가는 링크 없음. 역방향도 없음
- 기존 `sitemap.xml`에 `/search/` URL을 그대로 추가만 함 (같은 사이트맵 사용)
- `robots.txt` 수정은 필요한 경우에만 최소한으로

## 7. 첫 발행 범위 (MVP)

구현 단계에서 만들 최소 결과물:

- `search/assets/style.css` — 공통 스타일
- `search/template.html` — 새 글 작성용 복사 원본, 실제 제품 대신 플레이스홀더 포함
- `search/index.html` — 빈 허브 페이지. 글 추가 시 수동으로 목록 항목 추가
- **샘플 페이지 1개** — 예: `search/sample-top3.html`. 템플릿이 실제로 동작하고 SEO 요소(title, meta, canonical, OG, JSON-LD, 고지문구, sponsored nofollow 등)가 모두 들어있는지 검증용. 플레이스홀더 제품 데이터 사용
- `sitemap.xml` 업데이트 — 위 URL들 추가

## 8. 비범위 (Non-goals / YAGNI)

다음은 이 설계에 포함되지 않는다.

- 실제 카테고리 콘텐츠 작성 (첫 실제 글은 사용자가 카테고리를 정한 뒤 별도 진행)
- 카테고리 자동 태깅·검색 기능
- 댓글·뉴스레터·애널리틱스 대시보드
- 다국어
- 쿠팡 파트너스 계정 가입·승인 절차 (사용자 외부 작업)
- 동적 가격·재고 연동 (가격은 본문에 범위로만 표기하거나 생략)
- 기존 인형뽑기 가이드와의 교차 링크·공통 네비게이션

## 9. 열린 질문 (Open Questions)

스펙 승인 후 구현 계획 단계에서 결정:

- 샘플 페이지의 플레이스홀더 카테고리를 무엇으로 할지 (샘플이므로 가상의 제품명도 무방)
- `partners.js` 클릭 트래킹을 MVP에 포함할지, 이후 이터레이션으로 미룰지
- `index.html` 허브 페이지의 목록 갱신을 완전 수동으로 둘지, 아주 단순한 JS로 하드코딩된 배열에서 렌더링할지
